# Ресёрч: Алгоритмы процедурной генерации подземелий для roguelike

> Контекст проекта: Phaser 3, тайлы 16×16 (×3 = 48px на экране), 5-8 комнат на этаж, топ-даун 2D.

---

## 1. Обзор подходов

### 1.1 BSP (Binary Space Partitioning)

**Как работает:**
Рекурсивно делим карту на два прямоугольника, снова делим каждый — до нужной глубины. В конечных листьях дерева размещаем комнаты. Коридоры соединяют «сестёр» по дереву.

**Плюсы:**
- Гарантированное количество комнат (глубина дерева = кол-во комнат)
- Никаких перекрытий
- Простая реализация (100-150 строк TypeScript)
- Легко контролировать размер/форму комнат

**Минусы:**
- Все комнаты прямоугольные
- Коридоры прямые, «тюремный» вид
- Слабая органичность — всё слишком чётко

**Подходит нашему проекту?** Да — самый быстрый способ получить 5-8 комнат без перекрытий.

---

### 1.2 Random Room Placement (текущий подход)

**Как работает:**
Случайно размещаем комнаты, проверяем AABB-пересечения, отбрасываем конфликты. Соединяем L-образными коридорами.

**Плюсы:**
- Простота
- Быстро работает
- Уже реализовано

**Минусы:**
- Плотность размещения падает с ростом количества попыток
- Нет гарантии связности (нужен flood-fill или умное соединение)
- Коридоры могут пересекаться некрасиво

**Итог:** Работает для 5-8 комнат. Текущая реализация достаточна.

---

### 1.3 Delaunay Triangulation + MST

**Как работает:**
1. Генерируем много комнат со случайными позициями (physics separation или grid relaxation)
2. Строим граф Delaunay по центрам комнат
3. Находим минимальное остовное дерево (MST, алгоритм Краскала/Прима)
4. +10-15% случайных рёбер из Delaunay для образования петель
5. Прокладываем коридоры вдоль рёбер MST

**Плюсы:**
- Органичная связность
- Петли делают навигацию нелинейной
- Контролируемая структура (MST = минимум коридоров)

**Минусы:**
- Нужна реализация Delaunay (Bowyer-Watson, ~200 строк)
- Сложнее отлаживать
- Избыточно для 5-8 комнат

**Подходит нашему проекту?** Скорее нет — оверкилл для текущего масштаба.

---

### 1.4 Cellular Automata (пещеры)

**Как работает:**
Заполняем карту случайными стенами (~45-55%), затем несколько итераций: клетка становится стеной если ≥5 из 8 соседей — стены. Результат — органичные пещеры.

**Плюсы:**
- Очень органичный вид
- Пещеры без чётких комнат

**Минусы:**
- Нет гарантии связности (нужен flood-fill)
- Нет «комнат» — плохо ложится на нашу структуру
- Сложно контролировать кол-во и расположение

**Подходит нашему проекту?** Нет. Мы строим dungeon с комнатами, не cave.

---

### 1.5 Brogue-style (Room Addition)

**Как работает:**
Начинаем с одной комнаты. Итеративно добавляем комнаты, «пристыковывая» их к существующим через двери/коридоры. Комнаты сами по себе сделаны через cellular automata (органичные формы). После — добавляем «петли»: ищем стены между разными комнатами, пробиваем проход если расстояние по графу > 2.

**Плюсы:**
- Очень разнообразные формы комнат (прямоугольные + органичные)
- Гарантированная связность
- Петли создают нелинейность

**Минусы:**
- Сложная реализация
- Комнаты разных форм требуют другого подхода к тайлсетам

**Подходит нашему проекту?** Интересно в перспективе. Пока избыточно.

---

### 1.6 Binding of Isaac style (Grid BFS)

**Как работает:**
Сетка 9×8. Начинаем с центральной ячейки. BFS-расширение: добавляем соседнюю ячейку если у неё не более 1 соседа уже в списке (предотвращает петли и слишком широкие пути). Специальные комнаты (босс, магазин) — в самых дальних точках.

**Плюсы:**
- Идеально для «комнаты соединены дверьми» дизайна
- Очень простая реализация
- Структура с боссом в конце «встроена»

**Минусы:**
- Все комнаты одинакового размера (одна ячейка = одна комната)
- Нет коридоров — только двери между комнатами
- Не подходит для свободного тайл-пространства

**Подходит нашему проекту?** Нет. Наш проект — открытое тайловое пространство, не сетка комнат.

---

### 1.7 Enter the Gungeon style (Prefab Flows)

**Как работает:**
Заранее написанные «flow»-графы (макет уровня). Выбирается один из нескольких flow для этажа. Комнаты — пресозданные prefab-ы, вставляются в точки графа. Небольшая рандомизация в выборе prefab-ов.

**Плюсы:**
- Полный контроль над пейсингом
- Каждый run чувствует себя «спроектированным»

**Минусы:**
- Огромная работа по созданию контента
- Мало реиграбельности без большой библиотеки prefab-ов

**Подходит нашему проекту?** Нет на данном этапе.

## 3. Рекомендация для нашего проекта

**Краткосрочно (уже работает):**
Текущий Random Placement достаточен. Добавить flood-fill проверку связности — и можно считать алгоритм завершённым.

**Среднесрочно (следующий апгрейд):**
Перейти на **BSP** — даёт гарантированное число комнат (5-8), никаких перекрытий, простая реализация на TypeScript. Коридоры соединяют «сестёр» по дереву — читаемая структура уровня.

**Долгосрочно (если нужно разнообразие):**
**Brogue-style Room Addition** с cellular automata-комнатами — органичные формы, петли. Требует переработку тайлсетов для поддержки нестандартных форм комнат.

---

## 4. Что стоит добавить к текущему алгоритму

Без смены алгоритма можно улучшить текущую реализацию:

1. **Flood-fill проверка** — убедиться что все комнаты достижимы от старта
2. **Гарантированный коридор** — если комнаты изолированы, принудительно соединить ближайшую пару
3. **Специальные комнаты** — стартовая (без врагов), комната с боссом (последняя по удалённости от старта), комната с сокровищем
4. **Петли** — добавить 1-2 лишних коридора между некомнатными комнатами для нелинейной навигации
